# 🛠️ 거대 컴포넌트(500+ 라인) 리팩토링 가이드라인

본 문서는 단일 파일이 500줄을 넘어가는 등 높은 복잡도를 가진 코드를 안전하고 효과적으로 리팩토링하기 위한 원칙과 단계별 가이드를 정의합니다.

---

## 💎 핵심 원칙 (Core Principles)

### 1. 무결성 (Zero Side Effects)
- **리팩토링 전/후의 동작은 100% 동일해야 합니다.**
- 기존 기능이 망가지거나(Regression), 의도치 않은 파동 효과(Side Effect)가 발생해서는 절대 안 됩니다.
- UI 랜더링 결과뿐만 아니라 성능(메모리, 속도)적인 측면에서도 저하가 없어야 합니다.

### 2. TDD 기반 검증 (Test-Driven Development)
- 한 번에 코드를 모두 뜯어고친 뒤 테스트하는 것이 아니라, **테스트 코드를 먼저 확보하고** 변경을 진행합니다.
- 분리해 낼 로직이나 UI에 대한 유닛 테스트(Unit Test)를 작성하여, 변경 전후로 테스트를 돌려 기능이 유지됨을 기계적으로 증명하면서 개발합니다.

### 3. 관심사의 분리: UI와 비즈니스 로직의 격리
- 컴포넌트(UI) 파일 안에는 순수한 **렌더링(JSX) 로직만** 남깁니다.
- 데이터 페칭(Fetching), 복잡한 상태 계산, 포맷팅, 데이터 변환 등의 비즈니스 로직은 **Custom Hook**이나 외부 **유틸리티 함수**로 철저히 분리합니다.
- UI 컴포넌트는 오직 '받은 데이터를 어떻게 예쁘게 그릴 것인가'만 고민해야 합니다.

### 4. 단일 책임 원칙 (SRP) 기반의 소형화
- 파일 하나는 **하나의 명확한 역할(기능)**만 수행해야 합니다.
- 500줄이 넘어간다는 것은 하나의 파일이 여러 가지 일을 하고 있다는 증거입니다.
- 연관된 기능끼리 묶어 더 작은 단위의 컴포넌트로 잘게 나눕니다. (예: `Header`, `List`, `DetailView` 등으로 분할)

---

## 🚀 추가 권장 사항 (Advanced Tips)

### 5. 상태 관리(State Management)의 파편화 방지
- 컴포넌트를 분리할 때 하위 컴포넌트로 Props가 계속 전달되는 'Prop Drilling' 현상이 발생할 수 있습니다.
- 연관된 상태들은 묶어서 `Context API`나 전역 상태 관리(Zustand 등)를 활용하여 응집도를 높여야 합니다.

### 6. 타입을 명확한 외곽으로 분리 (`types.ts`)
- 컴포넌트 상단에 거대하게 자리 잡은 `interface`나 `type` 정의들은 가독성을 해칩니다.
- 여러 컴포넌트에서 공유될 수 있는 도메인 타입들은 별도의 타입 파일(예: `types.ts` 또는 📁 `types/`)로 분리합니다.

### 7. 성능 보존 (렌더링 최적화 챙기기)
- 파일을 나누면서 불필요한 리렌더링이 번질 수 있습니다.
- 분리된 하위 컴포넌트가 부모의 상태 변화에 쓸데없이 영향받지 않도록 `React.memo`, `useMemo`, `useCallback`을 적절히 적용하여 성능 최적화를 병행합니다.

### 8. 단계적(Incremental) 리팩토링과 잦은 커밋 (Small Commits)
- 한 번에 대규모로 뜯어고치지 마십시오.
- **[A 컴포넌트 분리 -> 테스트 통과 -> 커밋]** 과 같이 작은 호흡으로 전진해야 합니다.
- 문제가 발생했을 때 언제든 즉시 되돌아갈 수 있는(Rollback) 세이브포인트(Savepoint)가 필수입니다.

## ⚠️ 안티패턴 및 치명적 실패 사례 주의 (Lessons Learned)
*최근 `LogViewerPane.tsx` 리팩토링 실패(Regression)에서 얻은 교훈입니다.*

### 9. 강결합된 DOM 이벤트의 성급한 분리 금지 (Event Listener Decoupling Risk)
- `wheel`, `scroll`, `mouseup`, `mousemove` 등 브라우저 Native DOM 이벤트와 밀접하게 연관된 로직을 단순히 줄 수를 줄이기 위해 성급하게 Custom Hook으로 빼내면 타이밍 이슈가 발생합니다.
- 특히 `passive: false`가 필요한 `e.preventDefault()` 로직이나, `useRef`로 가져온 DOM Element에 직접 `addEventListener`를 붙이는 로직은 **해당 DOM 요소를 렌더링하는 컨텍스트(컴포넌트) 내부에 유지**하는 것이 더 안전할 수 있습니다.

### 10. 비동기 훅(Async Hook)과 라이프사이클 충돌
- 훅 내부에서 `import('react').then(...)`과 같은 동적 임포트(Dynamic Import)를 사용해 `useEffect`를 주입하려 하면, React의 컴포넌트 마운트 라이프사이클과 어긋나면서 브라우저 전역 이벤트(`window.addEventListener('mouseup', ...)`) 관리가 꼬여 드래그 및 선택 기능이 마비될 수 있습니다. (Race Condition 유발)

### 11. 부작용(Side Effect) 누락과 의존성 배열(Dependency Array) 파괴
- 코드를 훅으로 분리하면서 기존에 연결되어 있던 `SetState`, `Context` 또는 `Prop` 콜백(예: `handleFontZoom`, `onLineClick`)의 레퍼런스가 유실되는 경우가 흔합니다.
- 리팩토링 후에는 기존 컴포넌트의 기능들이 분리된 훅의 `useEffect` 의존성 배열(deps)에 올바르게 전달되어 **데이터 변경 시 자동 렌더링(render())이 촉발되는지** 반드시 확인해야 합니다.

## 🛠️ 실전 리팩토링 워크플로우 (Step-by-Step)

1. **테스트 확보**: 대상 시스템의 동작을 커버하는 테스트 코드 작성 및 실행.
2. **타입/상수 분리**: 하드코딩된 상수(`const`)와 `interface`들을 외부 파일 추출.
3. **로직 분리(Custom Hook)**: `useState`, `useEffect` 및 파생 데이터(`useMemo`) 로직을 `use[Component]Logic.ts` 같은 훅으로 빼내기.
4. **Sub UI 컴포넌트 쪼개기**: `renderHeader()`, `renderBody()` 같은 내부 함수들을 완전히 독립된 `Header.tsx`, `Body.tsx` 파일로 추출하기.
5. **통합 및 테스트**: 메인 파일에서는 분리된 훅과 하위 컴포넌트를 조립(Assemble)만 하도록 만듦. 전체 테스트 구동.
6. **최종 리뷰 및 성능 체크**: UI가 깨짐 없이 렌더링되는지 확인하고, React DevTools 등을 이용해 리렌더링 성능이 떨어지지 않았는지 검사.
