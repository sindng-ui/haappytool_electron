## 🚨 최우선 해결 과제: Cold Start 지연 (5.5s vs 3s)

### 1. 현상 요약
- **증상**: 앱 실행 후 대형 파일(1.4GB, 1200만 라인)을 처음 열었을 때, 필터링(체크/언체크) 소요 시간이 **5.5초~6초**대 유지.
- **특이점**: 동일 파일을 닫고 **다시 로드(Reload)**하면, 그때부터는 동일 작업이 **3.0초~3.2초**로 약 2배 빨라짐.
- **가설**: 
    - 첫 로딩 시 서브워커의 WASM 인스턴스가 OS 캐시나 JIT 최적화 혜택을 덜 받았을 가능성.
    - 메인 워커와 서브워커 간의 초기 데이터 파이프라인(RPC) 예열 문제.

### 2. 조사된 내용 (2026.03.04)
- **WASM 예열**: 인덱싱 중 서브워커를 미리 스폰하도록 수정했으나 효과가 미비함.
- **로그 확인**: 모든 서브워커(11개)가 `[WASM]` 모드로 동작 중임을 확인. 
- **청크 소요 시간**: WASM 필터링 자체는 청크(25만 라인)당 약 0.8s ~ 1.2s 소요 중.
- **병목 의심**: 
    - `line.toLowerCase()` 및 `textEncoder.encodeInto`를 라인마다 1200만 번 호출하는 오버헤드.
    - 메인 워커에서 `readFileSegment` RPC를 통해 데이터를 청크 단위로 가져오는 I/O 대기 시간.

## 🛠️ 향후 개선 아이디어 (Next Steps)

### 1. WASM 벌크 필터링 (Bulk processing) ⚡
- **방법**: 라인 단위로 WASM을 호출하지 않고, **UTF-8 버퍼 전체**를 WASM에 넘겨서 WASM 내부에서 뉴라인(`\n`)을 찾으며 한 번에 필터링.
- **기대 효과**: JS <-> WASM 브릿지 호출 횟수를 1200만 번에서 청크 수(약 48번)로 획기적으로 줄임. 

### 2. 메인 워커 I/O 최적화
- **방법**: `readFileSegment` 시 50MB씩 읽어오고 있는데, 이를 서브워커가 직접 읽거나 메인 워커가 Read-ahead(미리 읽기)를 통해 다음 청크를 대기 없이 서브워커에 던져주도록 개선.

### 3. WASM 모듈 캐싱 강화
- **방법**: `WebAssembly.compileStreaming` 결과나 인스턴스를 더 공격적으로 재사용하거나, 첫 로딩 시 Dummy 데이터를 한 번 돌려 엔진을 완벽히 "예열"하는 로직 검토.
