# Analyze Diff 성능 개선 계획서 (보수적 접근) 🐧⚡

형님, Analyze Diff의 분석 속도를 올리기 위해 기존 로직에는 변화를 주지 않으면서 **순수하게 CPU 부하만 줄이는** 보수적인 최적화 제안을 드립니다.

## 🛠️ 주요 개선 포인트

### 1. Happy Combo 태그 사전 처리 (Pre-lowercasing)
- **현상**: 매 로그 라인 필터링 시마다 모든 규칙의 태그를 `toLowerCase()` 하고 있습니다. (수만 번 반복)
- **개선**: 워커 시작 시 1회만 태그를 소문자로 변환하여 캐싱합니다.
- **기대 효과**: 분석 대상 로그가 많을수록 로그 인덱싱 속도가 크게 향상됩니다.

### 2. Point Metrics 전용 카운터 사용
- **현상**: 캡 보존을 위해 `Object.keys(pointMetrics).length`를 반복 호출합니다. (일부 JS 엔진에서 비효율적)
- **개선**: `pointMetricsCount` 변수를 별도로 두어 즉시 증감합니다.
- **기대 효과**: 분석 루프 내부의 오버헤드를 제거합니다.

### 3. 정렬 가중치 사전 계산 (Pre-calculated Severity)
- **현상**: 정렬(`results.sort`) 과정에서 `Math.abs` 연산이 중복 발생합니다.
- **개선**: 정렬 직전에 각 결과 객체에 `_severity` 가중치 값을 부여하고, 정렬 시에는 이 값만 비교합니다.
- **기대 효과**: 결과 데이터셋이 클 경우 정렬 시간을 단축합니다.
