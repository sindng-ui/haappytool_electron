# 로그 라인 수 불일치 해결 계획서 🐧🛠️

## 1. 문제 원인 분석
형님, 형님 말씀대로 대용량 로그를 불렀을 때 간헐적으로 Notepad++와 라인 수가 '몇 줄'씩 차이가 나는 원인을 파악했습니다! 🚀

원인은 바로 **고아 CR(`\r`) 문자**입니다.
*   **Notepad++ (Scintilla 엔진)**: `\n` (LF), `\r\n` (CRLF) 뿐만 아니라, 짝이 없는 단독 `\r` (CR) 문자도 하나의 독립된 줄바꿈으로 인식합니다. (주로 로그 파일에서 프로그레스 바를 렌더링할 때 `\r`로 덮어쓰기 하는 과정에서 이런 고아 `\r`이 남습니다.)
*   **HappyTool 기존 로직**: V8 엔진의 초고속 C++ 내장 함수인 `indexOf(10)` (`\n` 탐색)만 사용하여 파싱 속도를 극대화했습니다. 따라서 `\n` 없는 단독 `\r`은 줄바꿈으로 취급되지 않아, 결과적으로 라인 수가 Notepad++보다 적게 카운트되는 현상이 발생했습니다.

## 2. 해결 방안
성능 저하를 최소화하면서 Notepad++와 100% 동일한 라인 카운트 기준을 적용하기 위해 파싱 알고리즘을 개선합니다.

### 워커 파싱 로직 개선 (`workers/LogProcessor.worker.ts`)
*   기존의 단순 `indexOf(10)` 루프를 **`\n`과 `\r`을 모두 추적하는 고성능 단일 패스(Single-pass) 스캐너**로 교체합니다.
*   `\r\n`이 연속해서 등장할 경우를 완벽히 통제하기 위해 청크(Chunk) 경계에 걸치는 `\r` 문자의 상태(State)를 다음 청크로 이월하는 `pendingCR` 로직을 추가합니다.
*   자바스크립트 TypedArray 루프는 V8 엔진에 의해 고성능 기계어로 JIT 컴파일되므로, I/O 속도 대비 눈에 띄는 성능 저하 없이 완벽한 라인 일치율을 보장합니다.
